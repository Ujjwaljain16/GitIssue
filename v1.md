# Unified Issue Graph — V1 System Overview

## 1. What We Are Building

We are building an **invisible, GitHub-native issue intelligence layer** that:

* Ingests issues reliably from GitHub
* Stores them in a normalized, queryable format
* Prepares a foundation for detecting duplicates and linking related issues
* Acts as the first step toward a **cross-platform Issue Graph**

### In one line:

> A **production-grade ingestion and memory system for issues**, designed to later evolve into a unified issue graph across tools like GitHub and Jira.

---

## 2. Problem We’re Solving (V1 Scope)

Today:

* Issues are **fragmented across repos and tools**
* Duplicate issues waste maintainer time
* There is **no shared memory layer** across issues

### V1 focuses on:

> Building a **reliable, lossless system that captures issue data correctly**

Not solving everything yet—just establishing **ground truth**.

---

## 3. What V1 Does (Scope)

### Included

* GitHub App for webhook ingestion
* Secure webhook verification (HMAC)
* Event-driven ingestion pipeline
* Queue-based processing (no data loss)
* Normalized issue storage
* Signal extraction (basic structure)
* Idempotent updates (no duplication/corruption)

### Not Included (yet)

* No duplicate detection UI
* No embeddings / ML
* No Jira integration
* No federation
* No external API

---

## 4. System Architecture

```text
GitHub App
   ↓ (Webhook)
FastAPI Ingress (secure, async)
   ↓
Redis Streams (event queue, consumer groups)
   ↓
Worker (normalization + processing)
   ↓
PostgreSQL (source of truth)
   ↓
Logs + Metrics
```

---

## 5. Core Design Principles

### 1. Event-Driven System

* GitHub is treated as an **event source**
* No polling
* All updates flow through webhooks

---

### 2. Lossless Ingestion

* Every event is captured
* No silent drops
* Retry-safe via queue

---

### 3. Idempotency

* Same event can arrive multiple times
* System guarantees:

  * no duplicates
  * no corruption

---

### 4. Out-of-Order Safety

* Events may arrive out of order
* Database ensures:

  * newer updates always win
  * older updates never overwrite

---

### 5. Schema-Enforced Normalization

* All data passes through strict validation (Pydantic)
* No unstructured ingestion

---

### 6. Store Raw + Processed Data

* Raw payload محفوظ (for reprocessing)
* Normalized data for queries

---

## 6. Tech Stack

### Backend

* **Python + FastAPI**

  * Async support
  * Clean validation (Pydantic)
  * ML-ready for future phases

---

### Queue

* **Redis Streams**

  * Consumer groups
  * At-least-once delivery
  * Lightweight and fast

---

### Database

* **PostgreSQL**

  * Strong consistency
  * Structured schema
  * JSONB support (raw payload)
  * Future: pgvector for embeddings

---

### Cache / Infra

* **Redis**
* Hosting: Railway / Render / AWS

## 7. Data Model (Core)

### issues table

* external_id (global reference)
* repo, issue_number
* title, body, clean_body
* labels, state
* timestamps
* raw_payload (JSONB)

---

### signals table

Stores extracted signals:

* error messages
* file paths
* stack traces

Normalized for efficient querying.

---

## 8. Data Flow (End-to-End)

### Step 1: Issue Created on GitHub

↓

### Step 2: Webhook Sent to System

↓

### Step 3: FastAPI Ingress

* Verifies HMAC signature
* Wraps event with metadata
* Pushes to queue

↓

### Step 4: Redis Stream

* Stores event
* Guarantees delivery

↓

### Step 5: Worker Consumes Event

* Filters event type
* Normalizes payload
* Extracts signals

↓

### Step 6: Database Upsert

* Inserts or updates issue
* Ensures no stale overwrite

↓

### Step 7: Logs + Metrics Updated

---

## 9. Security

* HMAC verification for all webhooks
* Raw body validation (no parsing before verification)
* No secrets in logs
* Controlled DB access

---

## 10. Reliability Guarantees

| Property            | Guarantee                     |
| ------------------- | ----------------------------- |
| Data Loss           | None (queue + retries)        |
| Duplicate Events    | Safe (idempotent upsert)      |
| Out-of-order Events | Safe (timestamp guard)        |
| Crash Recovery      | Safe (unacked messages retry) |
